# Другий bottleneck — конкуренція за рядок `Product` (`pessimistic_write`)

У `create()` продукти зчитуються з `lock: { mode: 'pessimistic_write' }` перед перевіркою стоку й оновленням — це правильно для консистентності, але **один гарячий SKU** перетворює паралельні `POST /orders` на чергу транзакцій на одному рядку в Postgres.

## Умови прогону (відтворювані)

- **API:** `node dist/main.js`, `http://127.0.0.1:3000`, `NODE_ENV=production` (як у локальному `.env`).
- **База:** Postgres з тим самим `.env` (`DB_*`); сток перед кожним сценарієм скидається SQL-ом `UPDATE products SET stock = 500 WHERE id IN (1,2,3,4)` (скрипт нижче).
- **Авторизація:** один `POST /auth/login`, далі всі `POST /orders` з `Authorization` + унікальний **`Idempotency-Key`** (UUID) на кожен запит.
- **Паралелізм:** `N = 20` одночасних `fetch` у `Promise.all`.
- **Сценарій A — hot SKU:** усі 20 запитів з `items: [{ productId: 2, quantity: 1 }]` (один рядок під lock).
- **Сценарій B — spread SKUs:** `productId` циклічно `1,2,3,4,1,…` — по 5 замовлень на кожен з чотирьох товарів, різні row locks, менше взаємного блокування.

Скрипт: [`scripts/perf-create-lock-contention.mjs`](../scripts/perf-create-lock-contention.mjs).

```bash
# з кореня Lecture21, API і Postgres уже запущені
LOCK_BENCH_CONCURRENCY=20 node scripts/perf-create-lock-contention.mjs
```

## Результат одного прогону (2026-04-13, локальний Mac / Docker Postgres)

| Метрика | Hot SKU (усі → product `2`) | Spread SKUs (цикл `1..4`) | Відношення (hot / spread) |
|--------|----------------------------|---------------------------|---------------------------|
| Час стіни `Promise.all` (ms) | **1021** | **324** | **~3.15×** повільніше |
| Throughput (ордерів / с) | **~19.6** | **~61.7** | **~3.15×** нижче |
| p50 латентності одного `POST` (ms) | **~667** | **~231** | — |
| p99 (ms) | **~996** | **~318** | **~3.13×** вищий хвіст |
| Помилки | 0 / 20 | 0 / 20 | — |

**Як читаю:** при тому самому `N` і тій самій машині «всі в один SKU» дає помітно вищу стіну часу й перцентилі — це узгоджується з тим, що Postgres серіалізує доступ до одного рядка `products`. Рознесення по різних `id` знімає частину черги (хоча залишаються пул з’єднань, CPU, Rabbit тощо).

## Що заважало зняти чесний A/B спочатку

Після `commitTransaction()` код робив `this.orderRepository.findOne(...)`, тобто брав **друге** з’єднання з пулу, поки `QueryRunner` ще не звільнений у `finally`. При багатьох паралельних `create` на один SKU пул (дефолтно невеликий) **заклинював**: усі з’єднання зайняті, а новий `findOne` чекав на вільне — прогін виглядав як «вічне зависання», хоча корінь був не лише в row lock.

Я виправила це **точково**: після commit читати замовлення через `queryRunner.manager.findOne(Order, { … })` на тому ж раннері, потім як і раніше `release()` у `finally`. Після цього скрипт стабільно завершується й цифри відображають саме конкуренцію за рядок + загальний overhead, а не артефакт пулу.

## Що б дивилась у проді на цьому шляху

- Throughput `POST /orders` **per product_id** (агрегати по SKU), latency p95/p99 на create.
- `pg_stat_activity` / wait events (`transactionid`, `tuple`) під піками промо.
- Черга на один товар: рознесення інвентарю, шардінг, черга замовлень на SKU, або (якщо бізнес дозволяє) оптимістичний підхід з retry — вже продуктові рішення за межами цього ДЗ.
