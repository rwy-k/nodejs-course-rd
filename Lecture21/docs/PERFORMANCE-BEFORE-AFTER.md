# Before / After

## Умови виміру (однакові для «до» і «після»)

- **Сервіс:** Orders API на хості, `node dist/src/main.js`, `http://127.0.0.1:3000`.
- **База:** Postgres на `localhost`, БД `nestjs_shop`; перед прогоном я підправила `stock` для товарів 1–2, щоб не ловити 409.
- **Сценарій:** один логін → один `POST /orders` (product id=1, qty=1) → **80×** лише `POST /orders/:id/request-payment` на те саме замовлення (Authorize в Payments ідемпотентний; навантаження саме на читання ордера + gRPC).
- **До:** у `requestPaymentForOrder` був **`findOne`** з JOIN по `items`, `items.product`, `user`.
- **Після:** **`findOrderForPaymentAuthorize`** — `SELECT` тільки `id`, `totalAmount`, `idempotencyKey`.
- Між зміною коду й прогоном «після» я чекала **~8 с**, щоб Nest встиг перезавантажитись.

## Таблиця

| Метрика | До | Після | Як я це читаю |
|--------|-----|--------|----------------|
| **p95** (`request-payment`, ms) | **9.60** | **10.26** | У межах шуму одного ноута; головна історія не тут. |
| **p99** (ms) | **52.99** | **14.25** | Хвіст скоротився приблизно в 3.7 рази — схоже на те, що без зайвих JOIN-ів рідше вилітають «важкі» запити. |
| **CPU** | ~0% у простій між запитами (`ps` для PID :3000) | ~0% | Burst короткий, CPU не був головним обмеженням; зріз не ловить пік усередині 80 запитів. |
| **RSS** | ~15 MiB після прогону | ~15 MiB | На цьому рівні різниці я не побачила. |
| **Event loop lag** | не міряла | не міряла | У репо не підключала APM / `monitorEventLoopDelay`. |
| **Помилки** | **0 / 80** | **0 / 80** | Сценарій не зламала — для мене це must-have. |
| **Throughput** (req/s, той самий цикл) | **145.65** | **166.95** | Порядку +14.6%; стіна 0.549 s → 0.479 s. |
| **K8s / cost** | без `resources` у маніфесті | requests + limits на orders-api, payments, worker | На **локальному** HTTP-прогоні цифри в таблиці не змінюються — це про **кластер**: щоб под не їв ресурс «як попало». |

## Висновок для себе

- По **performance** найкраще «продає» історію **p99** і **throughput** на ізольованому `request-payment`; **p95** майже не рухнувся — я б явно казала на рев’ю, що перцентилі реагують по-різному.
- По **cost/runtime** зміни в `k8s/deployments.yaml` не видно в цьому бенчмарку, але вони все одно **реальні** для деплою в Kubernetes.
